01 / Concept

This Is Not a Chatbot

A chatbot answers questions. An autonomous agent holds your context, monitors your priorities, executes tasks independently, and reaches out to you when something matters. It remembers everything. It acts without being asked.

Typical Chatbot

Reactive / Stateless
  • Forgets everything between sessions
  • Only responds when spoken to
  • No persistent memory across conversations
  • Generic personality, no context about you
  • Can't take action on external services
  • Single conversation thread
  • No concept of time or scheduling

Autonomous Agent

Proactive / Persistent
  • 3-tier memory: facts, buffer, semantic search
  • Sends morning briefs and EOD checks unprompted
  • 11,000+ embedded vectors from your actual history
  • Identity document defines personality, beliefs, limits
  • 10 tools: databases, search, sheets, APIs
  • Autonomous task queue with scheduled heartbeats
  • Dreams at 1am, screens Upwork jobs at 7am

04 / The Stack

Tech Choices and Why

Every tool was chosen for a specific reason. No "everyone uses X" decisions. The goal was: minimum dependencies, maximum reliability, cheapest possible operation.

Tool What It Does Why This One
TypeScript Language Strict mode catches bugs before runtime. Same language for tools, agent, and API calls.
Grammy Telegram bot framework Long-polling only - no webhooks, no exposed ports, no attack surface. Bot reaches out, nothing reaches in.
OpenRouter LLM gateway Single API key for Sonnet, Haiku, and DeepSeek. Prepaid credits with hard spending limits. Uses the OpenAI SDK - just change the base URL.
Supabase Structured database + file storage Free tier is generous. Postgres under the hood. JS client handles auth. Storage for file uploads with signed URLs.
Pinecone Vector embeddings + semantic search Free tier: 100K vectors. Has its own embedding API (Inference) so you don't need a separate embedding service.
node-cron Scheduled tasks Lightweight cron inside the process. No external scheduler, no extra service to manage.
Zod Schema validation for tools Every tool input is validated before execution. LLMs send messy JSON - Zod catches it.
Railway Deployment GitHub push = auto-deploy. Health checks + auto-restart. $5/month for always-on process.

06 / Memory Deep Dive

The Hardest Part to Get Right

Memory is what separates an agent from a chatbot. Three tiers, each serving a different purpose. Core facts for identity. Buffer for recent context. Semantic search for everything that happened more than 20 messages ago.

Tier 1 - Always Loaded
Core Facts
Supabase core_facts
Permanent key-value pairs about the user. Goals, people, preferences, project states. Injected into every system prompt - the agent always has these. Max 4000 tokens. Facts are extracted automatically after every conversation by a background Haiku call.
goal.primary: financial stability + house
person.holly.role: wife
preference.style: direct, no fluff
project.pharadoxa.phase: 2 (railway)
machine.primary: Mac Studio M4 Max 128GB
Tier 2 - Rolling Window
Buffer
Supabase buffer + buffer_summary
Last 20 full messages (user + assistant). When message 21 arrives, old messages get compressed into a summary by a cheap Haiku call. Summary preserved, old messages deleted. This gives recent conversational context without unlimited token growth. Compaction is fire-and-forget - never blocks the response.

pipeline optimization. Decided to add
Google Sheets tool. Russ approved
vault write expansion..."

buffer: [messages 6-25]
messages 1-5: compressed into summary
Tier 3 - Semantic Search
Vector Memory
Pinecone / 1024d / cosine
Every exchange gets embedded after the response. When a new message comes in, search for the top 3 most similar past conversations (threshold: 0.3). This is how the agent remembers things from weeks ago that the buffer already compacted away. Two namespaces: conversations (live exchanges) and knowledge (ingested documents).

07 / The Agent Loop

How a Message Flows Through the System

User sends a Telegram message. The agent assembles context, calls the LLM, checks if it wants to use a tool, executes the tool, feeds the result back, and repeats - up to 5 times. Then fires background tasks.

Input
User sends Telegram message
Text, photo, document, audio, or video
Security
Whitelist check
Only TELEGRAM_CHAT_ID can interact. Everyone else is silently ignored.
Context Assembly
Build system prompt
soul.md + user.md + facts + buffer + RAG + tools (9 layers)
Core Loop
Call LLM (OpenRouter / Sonnet)
Send assembled context + available tools. Wait for response.
Tool call in response?
No - Return text response to Telegram
Yes - Execute tool, feed result back to LLM, repeat (max 5x)
Safety checks
Iteration count > 5? Stop.
Input tokens > 100K? Stop.
Neither? Continue loop.
Response
Send final text to Telegram
Plain text only. No markdown headers, minimal bold, mobile-friendly.
Background (non-blocking)
Promise.allSettled
Add to buffer + maybe compact | Extract facts (Haiku) | Embed to Pinecone | Log spend to Supabase | Log to conversations
// The critical pattern: background tasks NEVER block the response
await Promise.allSettled([
  addMessageAndMaybeCompact(role, content),
  extractFacts(exchange),  // Uses Haiku (cheap)
  embedToPinecone(exchange),
  logSpend(model, inputTokens, outputTokens, cost),
  logConversation(role, content, tokens, model),
]);
// Each task has its own try/catch. One failure doesn't kill the rest.
08 / Tools

10 Tools the Agent Can Use

The LLM decides when to call tools. Each tool has a Zod schema that validates input before execution. If the LLM sends bad JSON (and it does), Zod catches it and returns a clear error message.

10 / Gotchas

10 Things That Will Bite You

Every one of these cost us time. Some cost money. Learn from our pain.

Gotcha 01
Unbounded database reads will bankrupt your agent
Pharadoxa read 267 files from Supabase in one agent loop. 680K input tokens. $2.05 for a single message. Exhausted all OpenRouter credits instantly.
Fix: Cap tool output at 30K chars. Truncate content fields at 500 chars. Set a 100K token budget on the agent loop - stop iterating if exceeded.
Gotcha 02
Stale facts override current reality
The fact extractor wrote "tool.http_fetch.status: pending" once. That fact persisted for days, causing the agent to think it couldn't use http_fetch even though it was live and working.
Fix: Protect critical fact prefixes (tool.*, system.*, deployment.*) from LLM extraction. Only update these via direct database writes. Run weekly memory audits to clean stale facts.
Gotcha 03
Adding a tool without updating the system prompt
Built google_sheets tool, deployed it, but forgot to update system-prompt.ts. Pharadoxa didn't know the tool existed. LLMs can't introspect their own tool availability - they only know what the prompt tells them.
Fix: Rule: new tool = system prompt update in the same commit. No exceptions.
Gotcha 04
Promise.allSettled creates race conditions with async ops
addMessage and maybeCompact ran as Promise.allSettled siblings. With sync SQLite this was fine - with async Supabase, compaction checked the count before the insert completed. Buffer compacted with 19 messages instead of 21.
Fix: Chain sequential dependencies. addMessage first, then maybeCompact. Only use Promise.allSettled for truly independent operations.
Gotcha 05
OpenRouter credit exhaustion kills ALL models
Sonnet and Haiku use the same API key. When credits ran out, both died simultaneously. There's no "fall back to cheaper model" - they share a wallet. 402 error, hard stop.
Fix: Monitor credits proactively. Fail fast on 402 with a clear "top up credits" message. Don't retry - it won't help.
Gotcha 06
Pinecone dimension mismatch
Created the Pinecone index with 1536 dimensions (OpenAI ada-002 default). But multilingual-e5-large uses 1024 dimensions. Every embed would have failed silently or thrown cryptic errors.
Fix: Verify dimension match before building anything. Delete and recreate the index if wrong. This is a 30-second fix caught in pre-build, or a 2-hour debug caught in production.
Gotcha 07
Telegram markdown rendering is unreliable
Sent responses with markdown headers (##), bold (**), and bullet points. Telegram rendered them inconsistently across devices. Some messages were unreadable.
Fix: Plain text only. No markdown headers. Minimal bold. If you need structure, use line breaks and dashes. Mobile readability is what matters.
Gotcha 08
Supabase JS client can't run DDL
Tried to CREATE TABLE via the Supabase JS client. It doesn't support DDL statements. You need the dashboard, psql, or the MCP server for schema changes.
Fix: Create tables via Supabase dashboard, MCP, or raw SQL. The JS client is for DML only (select, insert, update, delete).
Gotcha 09
tsx -e doesn't work with ESM imports
Tried to run inline TypeScript with tsx -e "import { ... }". ESM imports fail in eval mode. Wasted time on a non-issue.
Fix: Always write a temp .ts file and run it with npx tsx temp.ts. Never use -e for anything with imports.
Gotcha 10
Batch ingestion without throttling hits rate limits
Pinecone free tier: 250K tokens/minute for inference. Batch embedding 5000+ documents blew through this instantly. Requests started failing silently.
Fix: Build three things into every batch script from day one: throttle delay between batches, retry with exponential backoff, and resume capability (track last successful batch).

The "How Do I Even..." Parts

Step-by-step for every external service. Click to expand.

  1. Open Telegram, search for @BotFather
  2. Send /newbot
  3. Choose a display name (e.g., "Pharadoxa")
  4. Choose a username ending in "bot" (e.g., "pharadoxabot")
  5. BotFather gives you a token - save this as TELEGRAM_BOT_TOKEN
  6. Send any message to your new bot
  7. Visit https://api.telegram.org/bot<TOKEN>/getUpdates
  8. Find your chat ID in the response - save as TELEGRAM_CHAT_ID
  9. Use this chat ID as a whitelist - ignore all messages from other IDs

Why long-polling over webhooks? Webhooks require an exposed URL. Long-polling means the bot reaches out to Telegram's servers - nothing reaches in. No ports to open, no SSL certs to manage, no attack surface.

  1. Create account at pinecone.io (free tier: 100K vectors)
  2. Create an index:
    • Name: your-project-name
    • Dimensions: 1024 (for multilingual-e5-large)
    • Metric: cosine
    • Region: us-east-1 (or closest)
  3. Copy your API key - save as PINECONE_API_KEY
  4. Install: npm i @pinecone-database/pinecone

The embedding trick: Pinecone has its own Inference API. Instead of paying for OpenAI embeddings separately, use pinecone.inference.embed() with the model "multilingual-e5-large". It's built into your Pinecone plan - no extra cost.

Critical: Match your index dimensions to your embedding model. ada-002 = 1536d. multilingual-e5-large = 1024d. Get this wrong and everything fails silently.

  1. Create project at supabase.com (free tier: 500MB database, 1GB storage)
  2. Go to Settings > API to get your project URL and keys
  3. Save SUPABASE_URL and SUPABASE_SERVICE_KEY
  4. Create tables via the SQL editor (Table Editor works too, but SQL is faster)
  5. Install: npm i @supabase/supabase-js

Service key vs Anon key: The anon key respects Row Level Security (RLS). The service key bypasses it. For a personal agent where you control all access, use the service key. For anything multi-user, use anon key + RLS policies.

Important: The JS client can NOT run CREATE TABLE or other DDL. It's for select/insert/update/delete only. Create your schema via the dashboard SQL editor first.

  1. Create account at openrouter.ai
  2. Go to Settings > Credits - add prepaid balance ($10 is plenty to start)
  3. Go to Settings > Keys - create an API key
  4. Save as OPENROUTER_API_KEY
  5. Install: npm i openai (yes, the OpenAI SDK)

Cost tip: Use Sonnet ($3/$15 per M tokens) for main conversations. Use Haiku ($0.80/$4 per M) for background tasks like fact extraction and summarization. A typical message costs $0.04-0.08.

  8. Push to GitHub - Railway auto-deploys
  9. Verify: close laptop lid, send Telegram message, get response

The gate test: Your agent must respond with your development machine completely closed/off. If it does, it's truly autonomous. If it doesn't, something is still depending on your local machine.

Docker tip: If you develop on Apple Silicon (arm64) but deploy to Railway (amd64), your Dockerfile needs --platform=linux/amd64. Native modules like better-sqlite3 will fail without this.
